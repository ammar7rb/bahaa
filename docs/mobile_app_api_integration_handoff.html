3. رحلة العميل من التسجيل حتى تفعيل الطلب

1
تسجيل العميل
إرسال terms_accepted: 1 وprivacy_accepted: 1 مع كل نوع تسجيل مستخدم.
2
التصفح والسلة
الضيف يتصفح ويضيف للسلة، لكن checkout يرجع customer_login_required حتى يسجل الدخول.
3
اختيار الشحن
تحميل three-step-options، وعرض العناصر التي available=true. Same Day يختفي بعد cutoff ويتحقق منه السيرفر.
4
فحص الباقة والـlimit
استخدام limit-summary وعدم حساب أسعار المنتجات أو shortage محليًا.
5
دفع المنتجات
أول طلب قد يصبح paid لكنه activation_pending حتى دفع فاتورة التأمين والباقة.
6
فاتورة التفعيل
بعد العودة من الدفع استدعاء current، وعرض package + insurance + total + الرسالة القادمة من السيرفر.
7
تحرير الطلب
بعد الدفع الرقمي أو موافقة الأدمن على offline: تتفعل الباقة، يمتد التأمين، ويصبح الطلب قابلًا للمعالجة.

أهم حقول القرار

الحقل / الكودتصرف التطبيق
customer_login_requiredفتح Login ثم إعادة checkout.
customer_purchase_package_requiredفتح شاشة الباقات.
customer_purchase_limit_insufficientعرض shortage واقتراح Extra Credit من assessment.
monthly_insurance_requiredفتح فاتورة التفعيل الحالية.
pending_offline_reviewعرض قيد مراجعة الإدارة ومنع أي تفعيل محلي.
قاعدة مالية: purchase limit يستهلك سعر المنتجات فقط. الشحن والضرائب لا تخصم من limit، وكل الحسابات تتم server-side.

4. رحلة التاجر والتدرج الإجباري للنشر

REGISTER -> VERIFY OTP -> ADMIN APPROVAL -> INSURANCE -> PACKAGE -> ADD PRODUCT -> PRODUCT APPROVAL -> PUBLISH -> OPTIONAL SEARCH / HOMEPAGE PROMOTION

التسجيل والمصادقة

  • التسجيل multipart/form-data ويحتاج بيانات المتجر والصور.
  • حفظ registration_reference حتى اكتمال OTP.
  • استخدام eligibility.next_step لفتح OTP أو شاشة انتظار الأدمن.
  • لا يسمح API v3 بالحساب الموقوف أو غير الموثق.

التأمين

  • يدفع مرة واحدة قبل شراء الباقة.
  • الدفع اليدوي يحتاج proof image حتى 5MB.
  • pending_review=true يعني انتظار الأدمن.

الباقة والحصص

  • لا تعتبر الباقة نشطة إلا مع status=active وpayment_status=paid.
  • استخدم remaining limits القادمة من السيرفر.
  • كل منتج جديد يحجز product quota مرة واحدة.
  • مدة الإعلان تبدأ عند أول نشر فعلي.

الترقيات

  • Search وHomepage حصتان مستقلتان.
  • اعرض زر الترقية فقط عندما can_promote=true.
  • لا تعِد ترتيب نتائج البحث أو عروض الصفحة الرئيسية محليًا.

5. الدفع اليدوي ومراجعة الأدمن

قاعدة إلزامية: أي Offline Payment لا يعني نجاح الدفع. التطبيق يعرض "قيد مراجعة الإدارة" فقط، ولا يفعّل طلبًا أو تأمينًا أو باقة قبل Approve من الأدمن.
العمليةإرسال التطبيقمكان مراجعة الأدمن
طلب العميلOffline order form + method informationOffline Orders داخل نظام الطلبات
فاتورة تفعيل العميلmethod_id + method_informationsCustomers -> Purchase Packages -> Offline Reviews
تأمين التاجرmultipart + payment_proofVendors -> Insurance Payments
باقة التاجرmultipart + package_id + payment_proofVendors -> Package Payments

State machine

pending / unpaid -> pending_review OR pending_offline_review -> paid / active (admin approve) -> rejected (admin reject)

Offline form configuration

  • يجب تفعيل إعداد offline_payment العام.
  • يجب أيضًا تفعيل طريقة Manual Transfer / Auto Payment Form.
  • الحقول الديناميكية تأتي من method_informations ولا تُكتب ثابتة في التطبيق.
  • تطبيق التاجر يرفع screenshot كملف multipart فعلي.

HTTP handling

HTTPالاستخدامتصرف التطبيق
200نجاح أو حالة business صحيحةاقرأ payload والحالة الداخلية.
401Token أو Login غير مسموحتسجيل الدخول أو إعادة OTP.
403Validation أو business ruleاستخدم code لتحديد الشاشة.
404عنصر أو طريقة دفع غير موجودةRefresh للقائمة.
422 / 429OTP غير صالح / انتظار resendعرض الرسالة والعداد.
الـparser يجب أن يدعم الشكلين: {message, code} و{errors: [{code, message}]}.

6. المطلوب على Android/iOS والفجوات الحالية

جاهز للربط الآن

  • تسجيل العميل مع الشروط والخصوصية.
  • منع checkout للضيف.
  • باقات العميل وlimit وextra credit الرقمي.
  • فاتورة التفعيل والدفع الرقمي.
  • الشحن الثلاثي وcutoff.
  • Homepage Featured Offers.
  • تسجيل التاجر وOTP وموافقة الأدمن.
  • تأمين التاجر الرقمي واليدوي.
  • باقات التاجر والحصص.
  • حماية إنشاء ونشر المنتجات.
  • Search Promotion وHomepage Promotion.
  • حماية API v2 وv3 للحساب غير المؤهل.

فجوات Backend قبل اعتماد كل الشاشات

1. Customer insurance status
لا يوجد endpoint مستقل GET /v1/customer/insurance/status. الحالة تُفهم من activation invoice ورفض checkout.
2. Effective vendor commission
seller-info يعيد override فقط؛ عندما يكون null لا يعيد النسبة العامة الفعلية.
3. Offline direct package / extra credit
الشراء المباشر لباقات العميل وextra credit يدعم digital فقط. Offline متاح لفاتورة التفعيل والطلبات.
4. Customer screenshot upload
فاتورة تفعيل العميل تتوقع قيمة payment_screenshot داخل JSON. لا يوجد multipart upload واضح للصورة الفعلية حتى الآن.

Checklist للمطور

□ فصل Customer token عن Seller token□ Polling / Refresh بعد الدفع والمراجعة
□ الاعتماد على code وليس نص message□ عدم تنفيذ optimistic activation
□ عرض Same Day فقط عندما available=true□ عدم حساب limit أو shortage محليًا
□ Badge للإعلان المميز في البحث□ Section للعروض المميزة في الرئيسية
الخلاصة العملية: الربط الأساسي جاهز لتطبيق العميل والتاجر. يبدأ مطور الموبايل بالمسارات الموجودة في صفحة Integration، ويؤجل الشاشات المرتبطة بالفجوات الأربع حتى استكمال Backend أو اعتماد صيغة واضحة لها.
